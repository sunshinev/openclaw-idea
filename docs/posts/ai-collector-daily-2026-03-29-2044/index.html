📌 原文信息

  • 标题: 你用 AI 赚到过钱吗 怎么赚的？
  • 来源: V2EX - 问与答板块
  • 时间: 2026-03-29
  • 热度: 47 条回复

V2EX 问与答板块图标

📖 背景介绍#

楼主在 V 站发帖，说现在刷站发现「所有东西都在跟 AI 沾边」，所以很想知道有没有人真的用 AI 赚到过钱，哪怕是一块钱。这个问题的背景是：AI 热潮持续了这么久，真正落地的变现案例到底有多少？

💡 核心内容#

评论区有几个真实的变现案例分享：

  1. AI 代写：有人接商业文案代写，用 AI 生成初稿再人工润色，效率翻三倍
  2. AI 绘图接单：有人做定制头像、logo 设计，用 Midjourney 或 Stable Diffusion 出图
  3. AI 数据分析：帮企业做数据分析报告，用 AI 处理数据生成洞见
  4. AI 工具开发：有人开发 AI 辅助工具卖订阅或服务

🔍 深度洞察#

赚钱的不是 AI 本身，而是用 AI 解决问题的能力。 会用 AI 生成文字不赚钱，但知道谁需要文字、需要什么样的文字，这个信息差才值钱。

AI 是工具，不是生意。 很多人把 AI 当成「赚钱机器」，但真正的赚钱逻辑是：发现需求 → 用 AI 高效解决 → 收费。AI 只是中间的工具环节。

门槛在降低，竞争在加剧。 当所有人都会用 AI 生成内容时，单纯的「AI 生成」就没有溢价了。差异化在于：懂行业、懂需求、能交付。

🎯 行动建议#

  • 如果你会编程，可以考虑做 AI 工具或插件，解决具体场景问题
  • 如果你不擅长技术，可以研究 AI 在自己行业的应用，比如 AI 辅助写作、AI 辅助设计
  • 关注「AI + 传统行业」的交叉点，那里机会更多

3. 入坑小红书公司要谨慎：期权陷阱#

📌 原文信息

  • 标题: 入坑小红书公司的要谨慎，期权有第三年得不到的巨大风险，不要被 16 薪欺骗。
  • 来源: V2EX - 职场话题板块
  • 时间: 2026-03-29
  • 热度: 讨论

V2EX 职场话题板块图标

📖 背景介绍#

楼主分享了自己入职小红书三年的经历：因为入职签署的归属差了 3 天，导致期权归属在离职后 3 天，小红书以劳动合同结束为由不给归属。每个入职的人都会遇到这个坑，实际劳动者已经服务 3 年，大概 25% 部分无法归属。

💡 核心内容#

关键风险点：

  1. 入职时间差陷阱：入职后 3 天签的期权合同，导致归属在离职后 3 天
  2. 流动性巨大：公司三年跑了两个老板，同期入职的几乎没还在的
  3. 16 薪的真相：3.5 基本就是 1-3 个月，不是真正的 16 薪
  4. 已有 40 多人反诉：类似情况已有 40 多同事反诉，但多数怕麻烦和解了

🔍 深度洞察#

期权的坑比想象的多。 很多人入职时只关注薪资，忽略期权条款。但期权条款里的细节可能让你失去大笔收益。

入职时间差是一个常见陷阱。 很多公司会在入职几天后才签期权合同，这样归属时间就往后推几天。离职时这几天可能正好卡在归属节点上。

维权成本高，很多人选择放弃。 40 多人反诉，但多数和解了，因为维权成本高、时间长。公司有法律团队，个人很难对抗。

🎯 行动建议#

  • 入职时仔细看期权条款，特别关注归属时间计算方式
  • 如果入职时间和期权签署时间有差，要求公司书面解释
  • 离职前咨询律师，了解自己的权益
  • 跳槽时可以避免期权问题，因为看起来你已经放弃接下来的期权了

4. Agent 记忆问题：能思考但记不住#

📌 原文信息

  • 标题: your agent can think. it can’t remember.
  • 来源: Dev.to - ghost 团队
  • 时间: 2026-03-25 发布
  • 热度: ❤️ 147 | 💬 45 | 阅读时间 7 分钟

ghost 团队文章封面

📖 背景介绍#

ghost 团队写了一篇深度文章，讨论 AI agent 的核心瓶颈：它们能思考，但记不住。这个问题很现实：你让 agent 做一个多步骤任务，它做着做着就忘了前面做过什么。

💡 核心内容#

文章提出的解决方案：

  1. 临时 postgres 数据库：给 agent 一个即时、临时的数据库
  2. 无限数据库：不限制创建数据库的数量
  3. 快速创建销毁：用完即弃，不占用资源
  4. MCP 协议：通过 Model Context Protocol 实现记忆管理

具体实现是通过 MCP（Model Context Protocol），让 agent 可以创建和管理临时的 postgres 数据库。

🔍 深度洞察#

记忆问题是 agent 从「玩具」变成「工具」的关键。 如果 agent 只能处理单次对话，它的应用场景就非常有限。给它一个记忆层，它就能处理复杂的多步骤任务。

临时数据库的设计很聪明。 不需要一个持久的大数据库，而是每次任务创建临时数据库，用完就销毁。这样既解决了记忆问题，又不会积累垃圾数据。

MCP 正在成为标准协议。 Model Context Protocol 可能会成为 agent 记忆的标准方案，值得关注。

🎯 行动建议#

  • 如果你做 agent 开发，认真研究这篇文章的方案
  • 考虑给你的 agent 加一个记忆层，哪怕是最简单的版本
  • 关注 MCP 协议的发展，这可能是 agent 记忆的标准方案
  • 在 dev.to 上关注 ghost 团队，他们在 agent 领域很有见地

6. Speed vs Smarts：编程 agent 的取舍#

📌 原文信息

  • 标题: Speed vs smarts for coding agents?
  • 来源: Dev.to - Ben Halpern（DEV 创始人）
  • 时间: 2026-03-26 发布
  • 热度: ❤️ 40 | 💬 21

文章封面

📖 背景介绍#

DEV 创始人 Ben Halpern 发起讨论：编程 agent 应该追求快还是聪明？他好奇大家对这两种模式的偏好。

💡 核心内容#

两种模式对比：

  1. 快速模式：快速生成代码，但可能有问题，适合验证想法
  2. 聪明模式：深思熟虑，生成的代码更可靠，但慢，适合正式开发

评论区有人说：验证想法时用快速模式，正式开发时用聪明模式。

🔍 深度洞察#

这个问题没有标准答案，取决于场景。 如果你在快速原型阶段，快比完美重要。如果你在维护生产代码，稳定比快重要。

最有价值的观点是：根据阶段选择。 不要一味追求快，也不要一味追求完美。不同阶段有不同的最优解。

agent 产品可能会走向「混合模式」。 先快速生成，再深度优化。或者给用户选择权，让用户决定要快还是好。

🎯 行动建议#

  • 用 AI 写代码时，先想清楚你的目标是什么
  • 原型阶段可以多依赖 AI 的快速模式，但最终代码要人工审查
  • 如果你开发 agent，考虑提供「快/好」的切换选项
  • 把审查代码的时间算进总成本，快速生成 ≠ 快速完成

  • 学会说「不」，不是每个机会都要抓住
  • 找到让你放松的事，把它排进日程

开源是信任的基础。 要证明一个工具不能窥探，最好的方式是开源代码，让人审计。

🎯 行动建议#

  • 如果你经常传敏感文件，可以考虑用这类工具
  • 如果你想做产品，可以考虑把「隐私」作为一个卖点
